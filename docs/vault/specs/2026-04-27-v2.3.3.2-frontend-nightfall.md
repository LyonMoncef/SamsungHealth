# V2.3.3.2 — Frontend Nightfall (auth pages + theme switcher)

## Vision

V2.3 → V2.3.3.1 ont livré toute la stack auth backend (register/login/refresh/logout, reset/verify, Google OAuth, rate-limit + soft backoff). Le frontend reste un dashboard data-viz mono-page (`static/index.html` + `dashboard.js`) **sans aucune UI auth**.

V2.3.3.2 livre le **frontend des flows auth** :
- 7 pages HTML statiques sous `static/auth/` (login, register, reset/verify request+confirm, oauth-link-confirm, oauth-callback handler).
- Premier consommateur de la rebrand spec [Data Saillance](2026-04-26-nightfall-rebrand-data-saillance.md) : tokens CSS `--ds-*` + theme switcher dark/light + bundle Playfair Display + Inter + 3 variantes logo.
- Vanilla JS (pas de framework) — cohérent avec `dashboard.js` existant.
- Backend wire-up : router FastAPI sert les pages statiques, route `/auth/login` → `/static/auth/login.html` etc.

**Pas de SPA** : chaque page est un HTML standalone qui POST en AJAX et redirige côté browser. Justification : 7 pages auth = simple, pas besoin de bundler/build step. Le dashboard data-viz reste séparé (pourrait évoluer vers SPA un jour, hors scope).

## Décisions techniques

### Architecture révisée post-pentester (3 HIGH bloquants → mitigations)

Le draft initial avait 3 frictions architecturales et plusieurs MED. Tous traités :
1. **`oauth-callback.html` flow ambigu** → clarifié : Google `redirect_uri` = `{PUBLIC_BASE_URL}/auth/google/callback` (backend GET, V2.3.2). Backend exchange code + valide id_token + retourne soit `200 JSON {access_token, refresh_token, ...}` soit `302` selon scénario. Le frontend N'a PAS de page `oauth-callback.html` séparée — V2.3.3.2 utilise un **redirect 302 backend** vers `/auth/oauth-success?return_to=...` (page intermédiaire qui lit les tokens depuis cookie httpOnly + sessionStorage selon décision token storage). Voir section "Flow OAuth callback révisé" plus bas.
2. **CSP `connect-src` Google endpoints inutile** → restreint à `connect-src 'self'`. Le frontend n'appelle JAMAIS Google directement.
3. **Security headers globaux** → middleware `SecurityHeadersMiddleware` appliqué **GLOBALEMENT** (X-Frame-Options DENY, X-Content-Type-Options nosniff, Referrer-Policy, Permissions-Policy, HSTS si HTTPS). CSP différencié par path (auth strict vs dashboard plus permissif vs API minimal).

Plus 6 MED critiques intégrés (cf. sections suivantes).

### Theme switcher (`theme.js`)

```js
// Default: respect prefers-color-scheme. User can override via toggle → persist localStorage 'ds-theme'.
const STORAGE_KEY = 'ds-theme';

function getEffectiveTheme() {
  const stored = localStorage.getItem(STORAGE_KEY);
  if (stored === 'light' || stored === 'dark') return stored;
  return window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light';
}

function applyTheme(theme) {
  document.documentElement.setAttribute('data-theme', theme);
}

function toggleTheme() {
  const next = getEffectiveTheme() === 'dark' ? 'light' : 'dark';
  localStorage.setItem(STORAGE_KEY, next);
  applyTheme(next);
}

// Apply ASAP to prevent FOUC
applyTheme(getEffectiveTheme());

// Bind toggle button
document.addEventListener('DOMContentLoaded', () => {
  const btn = document.querySelector('[data-theme-toggle]');
  if (btn) btn.addEventListener('click', toggleTheme);
});

// React to system theme change if user hasn't chosen
window.matchMedia('(prefers-color-scheme: dark)').addEventListener('change', () => {
  if (!localStorage.getItem(STORAGE_KEY)) applyTheme(getEffectiveTheme());
});
```

Le toggle est un bouton ☼/☾ en haut à droite de chaque page auth. Le dashboard existant ne sera pas migré ici (V2.3.3.2 = scope auth).

### `api-client.js` — error handling unifié

```js
const API_BASE = '';  // same-origin

async function apiPost(path, body, opts = {}) {
  const headers = {'Content-Type': 'application/json', ...(opts.headers || {})};
  if (opts.bearer) headers.Authorization = `Bearer ${opts.bearer}`;
  if (opts.regToken) headers['X-Registration-Token'] = opts.regToken;

  const r = await fetch(API_BASE + path, {method: 'POST', headers, body: JSON.stringify(body)});
  return _handleResponse(r);
}

async function _handleResponse(r) {
  let body = null;
  try { body = await r.json(); } catch {}
  if (r.status === 429) return {ok: false, status: 429, code: 'rate_limit_exceeded', retryAfter: r.headers.get('Retry-After')};
  if (r.status === 423) return {ok: false, status: 423, code: 'account_locked'};
  if (r.status >= 400) return {ok: false, status: r.status, code: body?.detail || 'unknown_error'};
  return {ok: true, status: r.status, body};
}
```

**Token storage** (révisé post-pentester) :
- **`refresh_token` → cookie httpOnly + Secure + SameSite=Strict + Path=/auth/refresh** (modif backend V2.3 ~10 lignes : `Set-Cookie` en plus du JSON sur `/auth/login`, `/auth/refresh`, `/auth/google/callback`). Cookie `Path=/auth/refresh` limite le scope aux endpoints qui en ont besoin. JS ne peut pas le lire → XSS ne peut pas exfiltrer la session longue.
- **`access_token` → `sessionStorage`** (15min TTL, court-vie, JS-readable car nécessaire pour `Authorization: Bearer` headers AJAX). Perte à fermeture onglet — UX acceptable car `/auth/refresh` automatique recrée un access depuis le refresh cookie.
- **Modif backend nécessaire** (à inclure dans V2.3.3.2) : `server/routers/auth.py::login`, `auth.py::refresh`, `auth_oauth.py::google_callback`, `auth_oauth.py::oauth_link_confirm` ajoutent `response.set_cookie("refresh_token", value=token, httponly=True, secure=ENV=="production", samesite="strict", path="/auth/refresh", max_age=30*86400)` en plus du JSON.
- **Modif `/auth/refresh`** : lit le refresh token depuis `request.cookies.get("refresh_token")` plutôt que body JSON (avec fallback body pour back-compat clients qui n'utilisent pas cookies).
- **Modif `/auth/logout`** : `response.delete_cookie("refresh_token", path="/auth/refresh")` en plus de revoke DB.

**Pourquoi maintenant et pas V3** : la migration plus tard avec users actifs = casser tous les refresh tokens existants. Faire le switch maintenant = 0 user impacté.

**`oauth-link-confirm.html`** :
- Lit `?token=` query param.
- Auto-submit on page load (POST `/auth/oauth-link/confirm`)
- 200 → store tokens reçus + redirect dashboard
- 400 → "Lien invalide ou expiré"

**Pas de `oauth-callback.html` côté frontend** (architecture révisée post-pentester #5).

**Flow OAuth callback révisé** :
1. Google redirige `GET /auth/google/callback?code=X&state=Y` directement vers le backend (pas vers le frontend).
2. Backend handler V2.3.2 traite : verify state, exchange code, validate id_token, account linking matrix.
3. Au lieu de retourner JSON, **le backend redirige 302 vers une page frontend selon le résultat** :
   - **Success** → set cookies httpOnly (refresh) + access_token en URL fragment **TEMPORAIRE** (`/auth/oauth-success#access_token=...&return_to=...`) → page `oauth-success.html` lit le fragment, store en sessionStorage, fait `history.replaceState({}, '', location.pathname)` pour purger l'URL, puis redirect `return_to` ou `/`. Le fragment n'est jamais loggué (browsers ne le mettent pas dans Referer ni server logs). C'est le pattern OAuth implicit flow (acceptable pour passage one-shot des tokens, pas pour storage long terme).
   - **Link pending** → 302 `/auth/oauth-link-pending.html` (page statique fixe "Email envoyé").
   - **Error** → 302 `/auth/oauth-error.html?code=oauth_user_declined|oauth_consent_required|...` (whitelist stricte de codes mappés, voir mapping V2.3.2).
4. La page `oauth-error.html` lit `?code=` query param, **whitelist stricte** des codes connus, affiche message FR figé. Aucun `error_description` jamais rendu.

**`oauth-success.html`** (nouveau, remplace `oauth-callback.html`) :
- Lit `#access_token=...&return_to=...` du fragment URL (pas query string).
- Store access_token en `sessionStorage` (refresh déjà en cookie httpOnly).
- `history.replaceState({}, '', location.pathname)` pour purger fragment de l'historique.
- Redirect `return_to` (validé contre whitelist côté frontend aussi) ou `/`.

**`oauth-error.html`** (nouveau) :
- Lit `?code=` query param.
- `_ALLOWED_OAUTH_ERRORS = {oauth_user_declined, oauth_consent_required, oauth_provider_error, oauth_provider_unavailable, oauth_callback_error, oauth_state_replay, oauth_id_token_invalid, oauth_nonce_mismatch, oauth_provider_email_unverified, oauth_registration_disabled, account_disabled}` → message FR figé.
- Si code inconnu → message générique "Erreur de connexion Google".
- **JAMAIS** afficher `?error_description=` ni autre query param libre.

**`oauth-link-pending.html`** (nouveau) :
- Page statique pure (pas de JS de read query). Texte fixe "Un lien de confirmation a été envoyé à votre email. Vérifiez vos messages." + bouton "Retour login".

### Anti-XSS frontend (pentester H1 + H2 fix)

**Règle stricte JS** : aucun usage de `.innerHTML =`, `document.write`, `eval`, `Function(...)`, `setTimeout(string, ...)`, `setInterval(string, ...)`. Uniquement `.textContent =`, `.innerText =`, ou DOM API `createElement` + `setAttribute` + `appendChild`.

**Test d'acceptation** : `tests/server/test_static_auth_pages.py::test_no_dangerous_js_patterns` — grep tous les fichiers `static/js/*.js` pour `.innerHTML`, `document.write`, `eval(`, `Function(`, refuse si trouvé. Tolérance UNIQUEMENT pour template literals avec valeurs hardcodées (i.e. `el.innerHTML = '<button>OK</button>'` literal sans variable runtime).

**Whitelist stricte des flash params** : tous les query params lus depuis l'URL pour pré-remplir l'UI (ex: `?registered=1` après register success, `?expired=1` après token expiry) doivent être validés contre une whitelist :

```js
const FLASH_PARAMS_WHITELIST = {
  registered: new Set(['1', '0']),
  expired: new Set(['1']),
  verified: new Set(['1']),
  reset: new Set(['1']),
};

function readFlashParam(name) {
  const params = new URLSearchParams(window.location.search);
  const raw = params.get(name);
  if (!raw) return null;
  return FLASH_PARAMS_WHITELIST[name]?.has(raw) ? raw : null;
}
```

### Login CSRF protection (pentester M1 fix)

**Backend modif** : `POST /auth/login`, `POST /auth/register`, `POST /auth/password/reset/request`, `POST /auth/verify-email/request` ajoutent un check `Sec-Fetch-Site` (déjà fait pour `/auth/google/start` en V2.3.2) :

```python
def _check_sec_fetch_site(request: Request) -> None:
    sfs = request.headers.get("sec-fetch-site")
    # Browsers modernes envoient toujours ce header. Allowed: same-origin, none (direct nav), absent (legacy/curl).
    # Refusé: cross-site, same-site (autre subdomain).
    if sfs in ("cross-site", "same-site"):
        raise HTTPException(403, detail="csrf_check_failed")
```

Évite : un attaquant qui force un user à login dans un compte malicieux via auto-submit form depuis un autre site (login-CSRF). Surface élargie de V2.3.2 OAuth-only à TOUS les POST auth.

- **`history.replaceState({}, '', location.pathname)`** dans `auth-form.js::onPageLoad` après lecture du `?token=` query param OU `#access_token=` fragment — purge l'URL de l'historique browser.
