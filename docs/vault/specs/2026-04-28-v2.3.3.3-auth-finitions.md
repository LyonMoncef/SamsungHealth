# V2.3.3.3 — Auth finitions

## Vision

V2.3 → V2.3.3.2 ont livré toute la stack auth + 9 pages frontend rebrand. **3 finitions restent** :

1. **Inter font bundle** — V2.3.3.2 utilisait fallback `-apple-system, sans-serif`. On bundle Inter local pour cohérence visuelle Data Saillance.
2. **Page admin UI** — V2.3 → V2.3.3.1 ont 4 endpoints admin (`GET /admin/pending-verifications`, `POST /admin/users/{id}/lock`, `POST /admin/users/{id}/unlock`, + nouveaux `GET /admin/users` + `GET /admin/users/{id}` ici). Self-host admin doit pouvoir les opérer **sans curl** via UI.
3. **Dashboard rebrand** — `static/dashboard.css` utilise des couleurs hardcodées `#hex`. Migration aux tokens `--ds-*` pour cohérence light/dark + theme switcher applicable.

V2.3.3.3 boucle l'auth complète. Suite logique : Phase 3 RGPD endpoints (`/me/export`, `/me/erase`) hors scope ici.

## Décisions techniques

### Architecture révisée post-pentester (3 fixes bloquants + 2 MED + 4 LOW)

**HIGH bloquants** :
1. **CSRF check `Sec-Fetch-Site` étendu à TOUS les POST admin** — V2.3.3.2 a oublié `admin.py` (lock/unlock V2.3.3.1). V2.3.3.3 corrige : `check_sec_fetch_site(request)` en première ligne de `lock_user`, `unlock_user`, `admin_probe`, et tout futur POST admin.
2. **`last_login_ip` HMAC dans `UserSummary` Pydantic** — actuellement INET brut leak en API admin (incohérent vs `auth_events.ip_hash` HMAC partout ailleurs). Fix : champ computed `last_login_ip_hash` calculé via `_ip_hash()` helper réutilisé. **Ne JAMAIS** exposer `last_login_ip` brut dans les responses JSON admin.
3. **`_cache_control_for` étendu à `/admin/*`** — bug latent : `security_headers.py` actuel applique `no-store` uniquement à `/auth/*`, donc pages admin restent dans cache navigateur après logout. Trivial.

**MED critiques** :
4. **Audit log à CHAQUE usage admin token** — `_check_admin_token` log `auth_event` `event_type='admin_token_used'` + `meta.endpoint` + `meta.ip_hash`. Forensique post-leak.
5. **Open redirect prevention `?return_to=`** — admin/login valide regex `^/admin/[a-z/-]+$` strict allowlist, sinon ignore.

**LOW + bonus** :
6. **`POST /admin/probe`** : second bucket global `100/hour` (anti-rotation IP via Tor) + `auth_event` log à chaque 401 (pas sample) + `asyncio.sleep(0.5)` constant sur 401.
7. **`request_id` field abuse refactor** — `lock_user` actuel met `reason+duration` dans `request_id` (V2.3.3.1). Refactor : déplacer vers `meta` JSONB column (déjà existe sur AuthEvent). Hors scope V2.3.3.3 — flag tech-debt NOTES.md.
8. **`dsColor()` sanitizer** : regex match `^(#[0-9a-fA-F]{3,8}|rgba?\(...\)|hsla?\(...\)|[a-z]+)$` ; fallback `#000000` + console.warn si pas color value valide.
9. **CSP admin `trusted-types` pour XSS hardening** : extension `CSP_AUTH_PAGES` pour `/admin/*` ajoute `require-trusted-types-for 'script'; trusted-types default;`.
10. **autocomplete admin token field** : `<form autocomplete="off">` + `<input type="password" autocomplete="off" autocorrect="off" spellcheck="false" name="admin-token">`.
11. **Dedup audit `admin_pending_verifications_read`** par fenêtre 5min (anti-spam log si admin garde onglet ouvert).
12. **Page Visibility API** : pause polling client-side si onglet pas visible.
13. **Token field UI label** : "Token admin" ou "Mot de passe administrateur" (PAS "X-Registration-Token" — pas leak du nom env var).
14. **`gitattributes binary`** : Inter TTF marqué binary pour éviter corruption Git.

### 2. Page admin UI

#### Endpoints backend nouveaux

`server/routers/admin.py` — 2 nouveaux endpoints :

```python
@router.get("/admin/users")
@limiter.limit("60/minute", key_func=_pure_ip_key)
async def list_users(
    request: Request,
    db: Session = Depends(get_session),
    _admin: None = Depends(require_admin_token),  # X-Registration-Token gated
) -> list[UserSummary]:
    """Liste tous les users (admin uniquement). Pagination simple offset/limit."""
    rows = db.execute(
        select(User)
        .order_by(User.created_at.desc())
        .limit(100)
    ).scalars().all()
    return [UserSummary.from_orm(u) for u in rows]


@router.get("/admin/users/{user_id}")
@limiter.limit("60/minute", key_func=_pure_ip_key)
async def get_user_detail(
    user_id: UUID,
    request: Request,
    db: Session = Depends(get_session),
    _admin: None = Depends(require_admin_token),
) -> UserDetail:
    """Détail user incluant identity_providers + auth_events récents (10)."""
    user = db.execute(select(User).where(User.id == user_id)).scalar_one_or_none()
    if user is None:
        raise HTTPException(404, "user_not_found")
    providers = db.execute(select(IdentityProvider).where(IdentityProvider.user_id == user_id)).scalars().all()
    recent_events = db.execute(
        select(AuthEvent).where(AuthEvent.user_id == user_id).order_by(AuthEvent.created_at.desc()).limit(10)
    ).scalars().all()
    return UserDetail(user=UserSummary.from_orm(user), providers=providers, recent_events=recent_events)
```

`UserSummary` Pydantic : `{id, email, is_active, email_verified_at, failed_login_count, locked_until, last_login_at, last_login_ip_hash, created_at}`. Le `password_hash` n'est **jamais** exposé. **Le `last_login_ip` brut n'est JAMAIS exposé** — uniquement `last_login_ip_hash` calculé à la volée via `_ip_hash(user.last_login_ip)` (helper réutilisé depuis `server/security/rate_limit.py`, HMAC-SHA256 + sel `SAMSUNGHEALTH_EMAIL_HASH_SALT` truncated 16 hex).

#### Auth admin via X-Registration-Token

V2.3 → V2.3.3.1 utilisent `X-Registration-Token` comme stop-gap admin auth. V2.3.3.3 garde ce pattern (pas d'admin role formel encore). Le frontend admin :
- Demande le token au load (sessionStorage `admin_token`, JAMAIS localStorage — perdu à fermeture onglet, sécurité accrue).
- Inject dans tous les fetch admin via header `X-Registration-Token`.
- Si 401 → redirect `/admin/login` (page nouvelle, simple form input password type=password).

`static/admin/login.html` (NEW) — simple form input "Token admin" + bouton "Connecter". Submit → POST `/admin/probe` (NEW endpoint qui retourne 200 si token valide, 401 sinon — permet de valider le token avant de l'utiliser dans le frontend).

#### Endpoint `POST /admin/probe`

`server/routers/admin.py` :
```python
@router.post("/admin/probe")
@limiter.limit("100/hour", key_func=lambda r: "global")  # second bucket global anti-rotation IP
@limiter.limit("10/minute", key_func=_pure_ip_key)
async def admin_probe(
    request: Request,
    _admin: None = Depends(require_admin_token),
) -> dict:
    """Sentinel endpoint pour validation client-side du admin token. Pas d'effet de bord.

    Pentester reco V2.3.3.3 : check_sec_fetch_site + sleep constant 0.5s sur 401 + audit
    chaque échec (pas sample 1/10).
    """
    check_sec_fetch_site(request)
    return {"status": "ok"}
    # Note : require_admin_token Depends raise 401 si token invalide. Le handler
    # _check_admin_token doit ajouter `await asyncio.sleep(0.5)` et `audit_event(...)`
    # AVANT raise 401 (forensique + anti timing).
```

### 3. Dashboard rebrand migration

`static/dashboard.css` actuel = couleurs hardcodées (gris foncé, accents néon). V2.3.3.3 migre aux tokens `--ds-*` :

```css
/* AVANT (hardcoded) */
.kpi-value { color: #FCBF0E; }

/* APRÈS (token) */
```

Ajout `<link rel="stylesheet" href="/static/css/ds-tokens.css?v=2.3.3.3">` dans `static/index.html` AVANT `dashboard.css` (cascade tokens d'abord).

Ajout du theme toggle dans le header dashboard (réutilise `static/js/theme.js`). Application `data-theme` sur `<html>` au boot (ssync FOUC prevention).

**Pas de changement structure dashboard.js** — uniquement les couleurs CSS migrent. Les viz D3/Canvas qui ont des couleurs hardcodées en JS (ex: `ctx.fillStyle = "#FCBF0E"`) doivent lire depuis `getComputedStyle(document.documentElement).getPropertyValue('--ds-accent-warm')`. Helper :

```js
// static/js/ds-colors.js (NEW)
export function dsColor(token) {
  return getComputedStyle(document.documentElement)
    .getPropertyValue(`--ds-${token}`)
    .trim();
}
```

`dashboard.js` : remplacer les `#hex` literals par `dsColor('accent-warm')` etc.

### CSP différencié `/admin/*` + Trusted Types (pentester #9)

`server/middleware/security_headers.py` étendu avec un CSP encore plus strict pour `/admin/*` :

```python
CSP_ADMIN_PAGES = (
    CSP_AUTH_PAGES                                         # base
    + " require-trusted-types-for 'script'; "              # tue les .innerHTML = userInput
    + "trusted-types default;"
)

if path.startswith("/admin/") or path.startswith("/static/admin/"):
    response.headers["Content-Security-Policy"] = CSP_ADMIN_PAGES
```

`Trusted Types` API (Chrome/Edge supporté, Firefox graceful fallback) bloque toute assignation à `.innerHTML`, `document.write`, etc., sauf si la valeur passe par une policy enregistrée. Notre admin frontend utilise UNIQUEMENT `.textContent` + DOM API → trusted-types compatible.

### Cache-Control `/admin/*` (pentester L3 fix)

`_cache_control_for` étendu :
```python
if path.startswith("/auth/") or path.startswith("/admin/"):
    response.headers["Cache-Control"] = "no-store"
```

Évite que pages admin restent en cache navigateur après logout (bug latent V2.3.3.2).

### Anti-IDOR sur `/admin/users/{user_id}`

Le `user_id` dans le path est un UUID v7. Sans auth, IDOR possible (`GET /admin/users/<random_uuid>`). Mitigation : `require_admin_token` Depends BEFORE le query → 401 sans token. Avec token, l'admin a accès à tous les users by design (c'est son rôle).

Pas d'endpoint `GET /admin/users/{user_id}` qui leak des données sans token.

### CSRF check `Sec-Fetch-Site` étendu admin (pentester HIGH H1 fix)

V2.3.3.2 a appliqué `check_sec_fetch_site(request)` aux endpoints `/auth/*` POST mais a oublié `/admin/*`. V2.3.3.3 corrige :

```python
@router.post("/admin/users/{user_id}/lock")
async def lock_user(user_id, body, request, ...):
    check_sec_fetch_site(request)  # HIGH H1 fix
    # ... reste
```

Appliqué à : `lock_user`, `unlock_user`, `admin_probe`, et tout futur POST admin.

### `request_id` field abuse (pentester H2 — flag tech-debt)

V2.3.3.1 `lock_user` met `request_id=f"reason:{body.reason}|duration_minutes:..."`. Mauvais usage du schéma. Les events `recent_events` retournés par `/admin/users/{id}` exposent ce champ → leak potentiel si admin met du PII dans `reason`.

**Fix V2.3.3.3** : sanitize `recent_events` côté API : si `event_type` ∈ `{admin_user_locked, admin_user_unlocked}`, on **strip** le `request_id` et on récupère `meta.reason` + `meta.duration_minutes` proprement (à condition que la colonne `meta` JSONB soit utilisée au lieu du `request_id` blob — refactor mineur dans `lock_user` à inclure dans V2.3.3.3).

### Open redirect prevention `/admin/login?return_to=` (pentester M1 fix)

```python
def _validate_admin_return_to(return_to: str | None) -> str | None:
    if not return_to:
        return None
    # strict allowlist : only /admin/{slug-with-dashes-and-slashes}
    if re.match(r"^/admin/[a-z0-9/-]+$", return_to):
        return return_to
    return None  # silently ignore unsafe value
```

### `dsColor()` sanitizer (pentester #8)

```js
const COLOR_RE = /^(#[0-9a-fA-F]{3,8}|rgba?\([^)]+\)|hsla?\([^)]+\)|[a-z]+)$/;

export function dsColor(token) {
  const raw = getComputedStyle(document.documentElement).getPropertyValue(`--ds-${token}`).trim();
  if (!COLOR_RE.test(raw)) {
    console.warn(`[ds-colors] invalid value for --ds-${token}, falling back: ${raw}`);
    return '#000000';
  }
  return raw;
}
```

### Polling pause via Page Visibility API (pentester #12)

```js
// pending-verifications.html JS
let pollInterval = null;
function startPolling() {
  if (pollInterval) return;
  pollInterval = setInterval(refresh, 30_000);
}
function stopPolling() {
  if (pollInterval) { clearInterval(pollInterval); pollInterval = null; }
}
document.addEventListener('visibilitychange', () => {
  if (document.hidden) stopPolling(); else startPolling();
});
startPolling();
```

Combiné avec dedup audit côté backend (5min window) → bruit log minimal.
